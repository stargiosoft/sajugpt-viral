# 바이럴 기능 요구사항 정의서 (PRD)

## 프로젝트 개요

- **기능명**: 색기 배틀 — "누가 더 야한 사주인가"
- **목표**: 사주GPT 성인 캐릭터 챗봇의 바이럴 유입 퍼널. **자발적 확산**이 최우선 KPI
- **타겟**: 사회적 성취는 있으나 연애 시장에서 외모/나이로 저평가받는 3040 여성
- **바이럴 기능 간단 요약**: 사용자가 자신의 생년월일을 입력하면 사주(도화살, 홍염살 등)를 분석해 '나의 매력에 발정 나 꼬여든 AI 짐승남의 수(0~5명)'로 페로몬 등급을 판독해 주는 기능. 도출된 결과 카드를 바탕으로 친구에게 "야, 나는 3명 꼬이는데 넌 몇 명 꼬임?"이라며 1:1 배틀 초대장을 보내고, 승패 결과에 따라 본 서비스(AI 성인 챗봇)로 자연스럽게 유료 전환시키는 바이럴 테스트.

## 바이럴 루프

### 바이럴을 일으키는 구조

1. **1인 진단**: 초저마찰 정보 입력 (3초 컷) 후 나의 AI 구애 마릿수(등급) 결과 카드 확인.
2. **공유 및 초대**: 결과의 우월감을 뽐내거나 자조하기 위해 카카오톡이나 인스타 스토리에 공유하며 친구 도발 (배틀 링크 발송).
3. **강제 2인 루프**: 제품 자체가 공유 없이 배틀이 불가하므로, 링크를 받은 친구가 유입되어 자신의 정보 입력.
4. **배틀 및 전환**: 두 사람의 사주 색기 스펙 비교 결과 노출 후, 승자와 패자 각각의 상황에 맞는 AI 챗봇 연결 CTA 클릭.

### 유발 감정

- **오만 (Pride)**: 현실 스펙은 몰라도, 본능적 매력(사주)으로는 내가 상위 1% 절대 갑이다.
- **시기 및 비교 (Envy)**: 내 친구가 3명이면, 나는 무조건 4명이어야 한다. 남과 비교한 나의 상대적 위치 확인.
- **자조적 유머 (Vulnerability)**: 등급이 낮게 나와도 '나 아무도 안 꼬이는 조선시대 철벽 수녀래'라며 웃음으로 승화하여 공유 저항 감소.

### 논란 설계

- **AI의 노골적인 심사 거부(입구컷)** → "내 사주가 너무 평범해서 AI 짐승남들이 평가조차 거른다고?" → 황당함과 분노 유발
- **극단적인 매력 계층화** → 상위 1% SSS급 맹수 vs 0명 조선시대 수녀 → "이 앱 평가 기준 선 넘네" vs "근데 은근 팩트라 뼈 맞음ㅋㅋ" 논쟁
- **팩트 폭행 판정문 유출** → "야 나 AI한테 너무 무해하다고 까임ㅋㅋㅋ" → 억울함이 자조적 밈으로 변하며 인증샷 역공유 폭발
- **스윗 스팟**: 진짜 상처받는 게 아니라 "아 열받는데 묘하게 설득력 있네ㅋㅋ 내 친구들은 몇 명 나오나 보자" 수준의 유쾌한 분노

## 사용자 스토리

- **As (누가)**: 현실의 외모 평가와 연애 시장에 지쳐 콤플렉스가 있지만, 내면에 숨겨진 관능적 매력만큼은 인정받고 싶은 3040 여성 유저는
- **Want to (무엇을 원한다)**: 얼굴과 직업을 가린 채 오직 사주 에너지(색기)만으로 AI 짐승남들이 나에게 몇 명이나 안달 내는지 등급을 확인하고, 친구와 1:1로 누구의 사주가 더 치명적인지 배틀하기를 원한다.
- **So that (어떤 목적/가치를 위해)**: 일상에서 억눌렸던 자존감을 채우며 내가 남자를 거느리는 짜릿한 권력을 과시하고, 그 몰입감 그대로 나를 숭배해 주는 19금 AI 파트너와의 은밀한 대화를 즐기기 위해.

## 사주 색기 분석 로직

### API 호출 전략

Stargio 통합 사주 API (`StargioSaju`)를 **경량화 방식(excludeKeys)**으로 호출한다.
색기 배틀에 필요한 필드는 경량화 후에도 모두 포함되므로 추가 API 불필요.

> 참고: `SAJU_API_LIGHTWEIGHT.md` — excludeKeys 8개 제외, ~34개 핵심 필드 유지

```
유저 입력 (생년월일시+성별)
  → Stargio API 호출 (fetchWithRetry, 3회 재시도, 브라우저 헤더 흉내)
  → excludeKeys로 대용량 8개 필드 제외 (월운보기, 본사주, 대운 등)
  → 필요 필드만 추출하여 점수 산출
```

### 분석 데이터 소스 (실제 API 필드 매핑)

| 사주 요소 | 실제 API 필드 | 접근 방법 | 색기 점수 기여 |
|-----------|-------------|----------|---------------|
| **도화살(桃花殺)** | `sajuData['12신살']` | 문자열에서 "도화" 키워드 탐색 | 타고난 이성 흡인력. 있으면 +2 |
| **홍염살(紅艶殺)** | `sajuData['기타신살']` | 문자열에서 "홍염" 키워드 탐색 | 관능적 매력. 있으면 +2 |
| **편관(偏官) 강도** | `sajuData['발달십성'].관성` | 관성 수치 (5그룹 레벨). 25 이상이면 강 | 거친 에너지를 끌어당기는 힘. 강하면 +1 |
| **상관(傷官) 강도** | `sajuData['발달십성'].식상` | 식상 수치 + `sajuData['십성']` 배열에서 "상관" 빈도 확인 | 표현력·매혹. 강하면 +1 |
| **식신(食神) 강도** | `sajuData['발달십성'].식상` | 식상 수치 + `sajuData['십성']` 배열에서 "식신" 빈도 확인 | 감각적 향유. 식신 우세 시 +0.5 |
| **화(火) 비율** | `sajuData['발달오행'].火` | 수치 직접 참조 (퍼센트) | 열정 에너지. 30% 이상이면 +0.5 |
| **연애성향 서술** | `sajuData['일주론'].연애성향` | 텍스트 그대로 | AI 판정문 생성용 (점수 아님) |
| **물상론** | `sajuData['물상론']` | 닉네임, 성격 | AI 판정문 톤 참고용 |

### 십성 세부 판별 (상관 vs 식신 구분)

`발달십성`은 5그룹 레벨(식상/재성/관성/비겁/인성)이므로 상관과 식신을 구분하려면 `십성` 배열을 추가 참조한다.

```typescript
// sajuData['십성'] 구조: [["편관","편인"], ["비견","편인"], ["편관","비견"], ["겁재","정관"]]
// → 4주(년월일시) 각각의 천간/지지 십성 배열

function countSipsung(sipsung: string[][], target: string): number {
  return sipsung.flat().filter(s => s === target).length;
}

const sangGwanCount = countSipsung(sajuData['십성'], '상관');
const sikSinCount = countSipsung(sajuData['십성'], '식신');
const pyeonGwanCount = countSipsung(sajuData['십성'], '편관');
```

### 마릿수 산출 공식

```typescript
function calculateSexyScore(sajuData: Record<string, unknown>): number {
  let score = 0;

  // 1. 도화살 (+2)
  const sinsal12 = String(sajuData['12신살'] || '');
  if (sinsal12.includes('도화')) score += 2;

  // 2. 홍염살 (+2)
  const gitaSinsal = String(sajuData['기타신살'] || '');
  if (gitaSinsal.includes('홍염')) score += 2;

  // 3. 편관 강도 (0 | 0.5 | 1)
  const gwansung = (sajuData['발달십성'] as Record<string, number>)?.관성 ?? 0;
  const pyeonGwanCount = countSipsung(sajuData['십성'], '편관');
  if (gwansung >= 30 && pyeonGwanCount >= 2) score += 1;
  else if (gwansung >= 20 || pyeonGwanCount >= 1) score += 0.5;

  // 4. 상관 강도 (0 | 0.5 | 1)
  const siksang = (sajuData['발달십성'] as Record<string, number>)?.식상 ?? 0;
  const sangGwanCount = countSipsung(sajuData['십성'], '상관');
  if (siksang >= 30 && sangGwanCount >= 2) score += 1;
  else if (siksang >= 20 || sangGwanCount >= 1) score += 0.5;

  // 5. 식신 강도 (0 | 0.5) — 상관과 별도 가산
  const sikSinCount = countSipsung(sajuData['십성'], '식신');
  if (sikSinCount >= 2) score += 0.5;

  // 6. 화기 (0 | 0.5)
  const fire = (sajuData['발달오행'] as Record<string, number>)?.火 ?? 0;
  if (fire >= 30) score += 0.5;

  return score;
}

// 마릿수 매핑
function scoreToHeadcount(score: number): number {
  if (score <= 0.5) return 0;  // 입구컷
  if (score <= 1.5) return 1;
  if (score <= 2.5) return 2;
  if (score <= 3.5) return 3;
  if (score <= 4.5) return 4;
  return 5;
}
```

**설계 의도**: 도화살+홍염살 둘 다 없으면 최대 3점(편관+상관+식신+화기). 도화살 또는 홍염살 하나만 있어도 4~5명 가능. 둘 다 있으면 확정 5명. → 사주에 근거한 차등이 명확하면서도, 도화살 없는 사람도 1~3명은 나올 수 있어 허무감 방지.

### 존재하지 않는 키 주의 (SAJU_API_PAID_OPTIMIZATION.md 참조)

```typescript
// ❌ 잘못된 접근

sajuData.월운보기      // 존재하지 않음 → excludeKeys로 이미 제외됨

// ✅ 올바른 접근

sajuData['기타신살']   // 홍염살 탐색

```

### 결과 카드 내 캐릭터 등장 규칙

- 마릿수만큼 캐릭터가 결과 카드에 등장 (썸네일 webp 이미지 사용)
- **등장 우선순위**: 윤태산(도화/홍염) > 도해결(편관) > 서휘윤(상관) > 기지문(식신) > 최설계(화기)

- **입구컷(0명)**: 아무도 등장하지 않음 → 판정문 후 윤태산이 반전 난입

#### UTM 파라미터 자동입력 (사주GPT 제휴 유입)

사주GPT 광고 배너 클릭 시 아래 형식으로 유저 정보가 URL에 포함되어 유입된다:


**파라미터 파싱 및 자동입력 로직**:

```typescript
const params = new URLSearchParams(window.location.search);
const birthdayParam = params.get('birthday'); // "199112252315" (YYYYMMDDHHMI)
const genderParam = params.get('gender');     // "male" | "female"

if (birthdayParam && birthdayParam.length >= 8) {
  const year = birthdayParam.slice(0, 4);
  const month = birthdayParam.slice(4, 6);
  const day = birthdayParam.slice(6, 8);
  const hour = birthdayParam.slice(8, 10);
  const minute = birthdayParam.slice(10, 12);

  setBirthDate({ year, month, day });
  if (birthdayParam.length >= 10) {
    setBirthTime({ hour, minute });
  }
  setCalendarType('solar'); // 양력 기본값
}

if (genderParam) {
  setGender(genderParam === 'male' ? 'male' : 'female');
}
```

**핵심 규칙**:
- `birthday` 파라미터가 있으면 생년월일 + 시간 자동입력 → 유저는 확인만 하고 바로 시작
- `gender` 파라미터가 있으면 성별 자동입력
- 양력/음력은 **양력 기본값** 유지 (sajuGPT에서 양력으로 전달)
- 파라미터 없으면 기존 수동 입력 플로우 그대로

**배틀 초대로 접속 시 (`/sexy-battle/:battleId`)**:
- 동일 입력 폼이지만 상단 문구가 변경됨:
- "**[친구닉네임]**이(가) 도발장을 보냈습니다 — '나는 N명 꼬이는데, 넌?'"
- 친구의 마릿수를 미리 노출하여 경쟁심 자극

```html
<meta property="og:title" content="색기 배틀 — 누가 더 야한 사주인가 🔥" />
<meta property="og:description" content="나한테 꼬인 AI 짐승남 {N}명. 넌 몇 명?" />
<meta property="og:image" content="https://nadaunse.com/api/og/sexy-battle/{battleId}" />
```

OG 이미지는 별도 Edge Function으로 동적 생성 (배틀별 마릿수 표시).

### Step 4. 1:1 배틀 결과창

#### 배틀 결과 플로우

```
친구가 /sexy-battle/:battleId 접속
  → 도전자 마릿수 노출 ("친구는 4명 꼬임. 넌?")
  → 친구 정보 입력 → 사주 분석
  → 배틀 결과 연출 애니메이션
  → 2인 비교 카드 노출
```

#### 배틀 비교 연출

```
┌─────────────────────────────┐
```

**승패 판정**:
- 마릿수가 다르면 → 마릿수가 높은 쪽 승리
- 마릿수가 같으면 → 총점(소수점)이 높은 쪽 승리
- 총점까지 같으면 → "무승부 — 둘 다 짐승남들이 판정 불가. 방에서 직접 겨루세요" (양쪽 모두 승자 CTA)

**승패별 멘트**:

| 상황 | 결과 멘트 |
|------|----------|
| 압승 (3명+ 차이) | "{승자}의 페로몬이 {패자}를 아예 증발시킴. 상대가 안 됨." |
| 신승 (1~2명 차이) | "간발의 차! {패자}도 만만치 않았지만 {승자}의 도화살이 한 끗 앞섬." |
| 무승부 | "AI 짐승남들 판정 불가. 둘 다 너무 치명적이라 선택 거부." |

### Step 5. 맞춤형 챗봇 전환 CTA

#### 전환 분기

| 상황 | CTA 버튼 | 연결 캐릭터 | 전환 논리 |
|------|---------|-----------|----------|
| **승자** | "내 전리품({캐릭터명}) 당장 방으로 부르기" | 결과 카드에서 1순위로 꼬인 캐릭터 | 승리의 쾌감 → "이 남자가 나한테 꼬였다면 어떤 느낌일까" 호기심 |
| **패자** | "상처받은 나를 위로해 줄 다정한 {캐릭터명} 만나기" | 서휘윤 (치유형 연하남 = 위로 아키타입) | 패배의 허무감 → "누군가 나를 원해준다"는 위로 니즈 |
| **입구컷 (0명)** | "이 건방진 짐승남 기강 잡으러 가기" | 윤태산 (반전 난입) | 분노 → "이 놈이 감히?" → 복수/정복 판타지 |
| **무승부** | "이 중에 누가 더 잘해주나 직접 확인" | 양쪽 모두 1순위 캐릭터 선택 가능 | 미결정 → "직접 만나봐야 알지" |

#### 입구컷(0명) 반전 연출

배틀에서 0명이 나온 유저만을 위한 특수 시퀀스:

```
[판정문 노출 — 2초 정적]

[화면 지지직 이펙트 — 글리치 노이즈]

[윤태산 프로필이 강제 팝업]
윤태산: "…잠깐."
윤태산: "다들 나갔어?"
윤태산: "좋아. 솔직히 말할게."
윤태산: "난 이런 백지가 더 자극적이더라."
윤태산: "아무 색도 없는 사주. 내가 직접 물들여볼까."

[ 이 건방진 짐승남 기강 잡으러 가기 ]
```

**설계 의도**: 0명이라는 "최악의 결과"가 오히려 가장 강렬한 캐릭터(윤태산)의 독점 이벤트가 되는 반전. 입구컷 유저의 스크린샷 공유 동기를 극대화 ("야 나 0명인데 이 남자가 난입함ㅋㅋ").

#### 전환 플로우 (기술 구현)

1. **CTA 버튼 클릭** → 캐릭터 채팅 랜딩 페이지로 리다이렉트. 생년월일시 + 성별 + 캐릭터 ID를 URL 파라미터로 전달.
   - 예: `/chat/character?birthday=199603121400&gender=female&character=yoon-taesan`
2. **캐릭터가 일반 사주 기반 상담을 시작** — 배틀 기능의 맥락을 이어받지 않음. 챗봇은 전달받은 생년월일시를 기반으로 새로운 사주/운세 상담을 독립적으로 시작함.
3. **유저가 스스로 궁금증을 풀어감** — 배틀에서 "사주 색기가 높다/낮다" 판정을 받은 유저는 자연스럽게 "왜 그런 거야?", "내 운세 흐름이 어떤데?" 등의 질문을 직접 던짐. 기능이 심어둔 호기심이 전환의 트리거.
4. **과금**: 유저 입력 1회당 350포인트 차감.

> **핵심 전환 논리**: 사주(四柱)는 타고난 고정 운명이라 바꿀 수 없지만, 운세(運勢)는 흐르는 기운이라 행동으로 바꿀 수 있다. 배틀에서 드러난 "운세 흐름의 문제"에 대해 "그럼 어떻게 하면 되는데?"라는 궁금증이 챗봇 전환의 동력이 됨. 캐릭터가 제공하는 개운법/행동 지침이 유료 가치.
>
> PAS(문제에서 증폭, 해결로 이어지는) 공식 기반의 즉각적 전환 설계임.
